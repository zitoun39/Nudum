# Nudum Module Boundaries

## Purpose

This document defines the interaction rules, communication patterns, and boundaries of modules within the Nudum platform.

It guarantees that business modules remain highly decoupled, allowing independent development and enabling smooth migration paths to microservices if required in the future.

---

## Architectural Principle

Every business capability and database table has exactly one owner module.
Direct cross-module database joins or queries are strictly prohibited.

---

## Module Boundary Integrity

```text
       [Mahattati Module]                      [Jawdati Module]
               │                                       │
      (Owns Sites/Stations)                  (Owns Laboratories)
               │                                       │
               ▼                                       ▼
       [Mahattati DB Schema]                  [Jawdati DB Schema]
               │                                       │
               └─────────  Cross-Context Read ─────────┘
                   (Via API Client or Domain Event)
```

No database query may span multiple schemas.

---

## Cross-Module Reference Patterns (Anti-Corruption Layer)

When an entity in Module A needs to reference an entity in Module B (e.g. Jawdati `Sample` collected at Mahattati `Station`), it must use one of the following **Anti-Corruption Layer (ACL)** patterns:

### 1. Loose Reference by ID
* **Rule**: Store only the foreign entity's UUID (e.g., `station_id: UUID`) in the local table.
* **Querying**: The client (React SPA) fetches the data by combining the records in the UI layer, calling the respective APIs independently.

### 2. Event-Time Data Snapshot (Denormalization)
* **Rule**: If the referenced data is immutable or must be preserved for historical compliance, store a snapshot of the required field value locally at the time of the action (e.g., `station_name_snapshot` inside the `laboratory_samples` table).
* **Benefit**: Ensures the sample sheet remains readable even if the station is renamed or deleted in Mahattati.

### 3. Local Read Models (Event Sourcing)
* **Rule**: If high-frequency querying is required, the consuming module subscribes to domain events (e.g., `StationUpdated`) and keeps a local, minimal read-only table of the foreign entity.
* **Benefit**: Zero runtime network dependencies or cross-module API lookups.

## Domain Event Communication

Business modules communicate asynchronously by publishing events to the internal event bus.
Events must describe facts in the past tense:

* **Mahattati**: `StationCreated`, `EquipmentInstalled`, `MaintenanceCompleted`.
* **Jawdati**: `SampleCollected`, `AnalysisCompleted`, `ResultValidated`.
* **Archivi**: `DocumentUploaded`, `DocumentApproved`.

Subscribers catch events and update their local states. The publisher does not know who consumes its events.

---

## Shared Platform Core Abstractions

Shared services are provided strictly by Core Infrastructure. Business modules access them via interfaces:
- **File Attachment**: Business modules store binary references (e.g. `file_id`) in their tables. They request presigned upload/download URLs from the Core File Service.
- **Workflow State**: Modules register transitions with the Core Workflow Engine. Business rules are validated in the module via event listeners (callbacks), keeping Core domain-free.
- **Auditing**: Audit logs are generated by publishing `AuditEventOccurred` events, letting Core handle formatting and persistence asynchronously.
